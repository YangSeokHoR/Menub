# menub — 메뉴바 허브 앱 개발 지침서

> **menub** — 여러 개인 유틸리티 앱이 각자 차지하던 메뉴바 아이콘을 한 자리에 모으는 단일 진입점. 핵심 그림은 "메뉴바를 팝오버 하나로 접는다".
> 이름은 필요 시 교체 가능. 이 문서는 cowork 프로젝트의 **기준 문서**이며, 아래 *확정 명세*는 합의 없이 임의 변경하지 않는다.

---

## 1. 프로젝트 개요

**한 줄 요약**
개인 유틸리티 앱들이 각각 메뉴바 아이콘을 하나씩 차지해 메뉴바가 넘치는 문제를, 아이콘 **딱 하나**로 모든 도구에 접근하는 허브 앱으로 해결한다.

**목적**
- 메뉴바 공간을 1개 아이콘으로 고정 — 도구가 늘어도 노치 뒤로 밀려 사라지지 않는다.
- 기존 위성 앱들을 새로 짜지 않고, 얇은 계약만 추가해 허브에서 통합 접근한다.
- "어떤 도구를 허브에 넣을지"를 사용자가 직접 켜고 끈다.

**가치 가설**
위성 앱이 자기 기능 목록만 표준 형식으로 노출하면, 허브는 그 목록을 자동으로 읽어 하나의 UI로 묶을 수 있다. 그러면 메뉴바 점유는 1로 고정되고, 위성 앱 추가/변경 시 허브 쪽 수작업이 사실상 사라진다.

## 2. 확정된 핵심 명세

| 항목 | 확정 내용 |
|---|---|
| 플랫폼 / 스택 | macOS 앱, Swift / SwiftUI (필요 시 AppKit 연동) |
| 빌드 방식 | 바이브코딩 (아래 §10 협업 규칙 적용) |
| 통합 모델 | **런처형** — 기존 위성 앱은 유지, 허브가 실행 / 토글 / 기능 호출만 담당 |
| 메뉴바 점유 | 허브가 상태 아이템 **1개**만 등록 |
| 기본 진입 UI | **팝오버 패널** (SwiftUI, 도구 그리드/리스트) |
| 보조 진입 UI | **전역 단축키로 뜨는 검색 팔레트** (전 위성 액션 통합 검색) |
| 상단 고정 | 팝오버에서 **Runlet을 맨 위에 핀 고정** |
| 커스텀 제약 (하드) | 각 도구를 허브에 **넣을지 말지 사용자가 토글**로 정한다 |
| 위성 아이콘 처리 | 허브에 등록된 위성 앱은 **자기 메뉴바 아이콘을 숨긴다** (해제하면 다시 표시) |
| 기능 접근 범위 | 허브 안에서 그 위성의 **원래 메뉴바 기능을 전부** 호출할 수 있어야 한다 |

### 협업자에게 위임한 열린 결정 (기본값으로 진행, 합의 시 변경)

| 결정 | 채택한 기본값 | 뒤집을 수 있는 대안 |
|---|---|---|
| 노출 메커니즘 | **매니페스트 자동 탐색 + URL scheme 호출** (발견=자동, 실행=URL) | 순수 수동 등록(URL scheme만, 목록은 사용자가 허브에 직접 입력) |
| 매니페스트/설정 저장 위치 | 공유 `~/Library/Application Support/menub/` 폴더 | App Group 컨테이너 |
| 샌드박스 | **비샌드박스** (개인용, App Store 배포 아님 → 임의 앱 실행·폴더 접근 목적) | 샌드박스 + 필요한 entitlement/보안 스코프 |
| 팝오버 2뎁스 네비게이션 | 도구 선택 → 같은 패널에서 액션 목록으로 전환 | 도구별 서브뷰/디스클로저 펼침 |
| 위성 상태 표시 범위 | 실행 중 여부 정도만 (경량) | 위성별 커스텀 상태 배지 |

## 3. 비목표 (범위 밖)

- 위성 앱을 허브 안 모듈로 **흡수/재작성**하지 않는다. (런처형 유지)
- 서드파티/타사 앱(내가 만들지 않은 앱)의 메뉴바 아이콘을 관리하는 **범용 메뉴바 매니저**가 아니다. Ice/Thaw/Bartender의 대체품이 아님.
- 접근성 API로 남의 메뉴바 아이콘을 대신 클릭하는 방식은 채택하지 않는다. (취약성 때문)
- 위성 앱의 **UI를 허브 안에 임베드(다른 프로세스 뷰 삽입)** 하지 않는다. 허브는 액션을 "호출"할 뿐 위성의 화면을 품지 않는다.
- 클라우드 동기화, 멀티 유저, 원격 제어는 범위 밖.

## 4. 기술 스택 & 환경

- **언어/UI**: Swift, SwiftUI. 상태 아이템은 SwiftUI `MenuBarExtra`(macOS 13+) 사용, 부족한 부분만 AppKit 연동.
- **Dock 아이콘 숨김**: `Info.plist`의 `LSUIElement`(Application is agent)로 허브를 메뉴바 전용 에이전트로 만든다.
- **위성 실행/URL 호출**: `NSWorkspace`로 앱 실행 및 URL 열기.
- **URL scheme**: 위성 앱은 `Info.plist`의 `CFBundleURLTypes`에 자기 scheme 등록, SwiftUI `onOpenURL`(또는 앱델리게이트)로 수신.
- **전역 단축키(팔레트)**: 공개 SwiftUI 전역 단축키 API가 마땅치 않음 → 구현 시 Carbon Hot Key(`RegisterEventHotKey`) 또는 전역 이벤트 모니터 중 택1. **방식·가용성은 구현 단계에서 확정**하고, 접근성 권한 필요 여부를 확인해 문서에 반영한다. (추측 시그니처 사용 금지)
- **저장소**: 공유 폴더 `~/Library/Application Support/menub/` 하위에 매니페스트 폴더와 허브 설정 파일을 둔다.
- **샌드박스**: 기본 **비샌드박스**. 임의 위성 앱 실행과 공유 폴더 접근을 위해서다. (App Store 배포로 방향이 바뀌면 §8-5의 트레이드오프 재검토)

> 버전 한정 API는 반드시 가용성을 명시할 것: `MenuBarExtra` / `MenuBarExtraStyle.window`는 macOS 13+, `onOpenURL`·`NSWorkspace.openApplication(at:configuration:)`은 macOS 11+.

## 5. 사용자 플로우

**첫 실행**
1. 메뉴바에 menub 아이콘 1개가 뜬다. Dock 아이콘은 없다.
2. 아이콘을 누르면 팝오버가 열리고, 감지된 위성 후보 목록과 등록 안내가 보인다.

**위성 등록 (허브에 넣기)**
1. 설정에서 감지된 위성 목록이 토글과 함께 표시된다. (매니페스트가 있는 앱이 후보로 뜬다)
2. 특정 위성을 켜면 → 허브 팝오버/팔레트에 그 위성과 액션들이 나타난다.
3. 켜진 위성은 **다음부터 자기 메뉴바 아이콘을 숨긴다.** (허브가 유일한 진입점이 됨)

**일상 사용 — 팝오버**
1. 메뉴바 아이콘 클릭 → 팝오버 오픈. **Runlet이 맨 위**, 그 아래 나머지 위성들.
2. 위성 선택 → 같은 패널에서 그 위성의 액션 목록(2뎁스)으로 전환.
3. 액션 클릭 → 해당 위성이 실행(필요 시)되고 그 기능이 수행된다.

**일상 사용 — 팔레트**
1. 전역 단축키 → 화면에 검색 팔레트 등장.
2. 모든 위성의 모든 액션이 하나의 평평한 목록으로 합쳐져 검색된다. (예: "runlet 배포")
3. 엔터 → 실행. 마우스 없이 완결.

**커스텀 (허브에서 빼기)**
1. 설정에서 위성 토글 off.
2. 그 위성은 허브 목록에서 사라지고, **다시 자기 메뉴바 아이콘을 표시**한다.

## 6. 아키텍처 — 모듈 구조와 의존 방향

```
menubApp (SwiftUI App)
└─ MenuBarExtra scene (.window 스타일)  ← 단일 상태 아이템 소유
   ├─ PopoverRootView          기본 진입 UI: 위성 그리드/리스트 (Runlet 핀 상단), 2뎁스 액션
   └─ SettingsView             위성 토글(허브 포함 여부), 핀/정렬, 단축키 설정

PaletteWindow (borderless panel)         보조 진입 UI: 전역 단축키로 호출, 전 액션 통합 검색

── 코어 (뷰 비의존) ─────────────────────────────
RegistryStore   매니페스트 폴더를 읽고 변경 감시 → [SatelliteManifest] 제공 (가용 도구의 진실원천)
ConfigStore     허브 자체 설정 저장/로드: 위성별 enabled·pinned·sortIndex, 팔레트 단축키
Invoker         SatelliteAction 실행: 필요 시 위성 앱 실행 후 invoke URL 열기 (NSWorkspace)
RegistrationCoordinator  위성 등록/해제 시 "허브가 관리함" 플래그를 공유 저장소에 기록 → 위성이 자기 아이콘 숨김/표시
HotkeyManager   전역 단축키 등록 → PaletteWindow 토글 (구현 방식 §8-1에서 확정)
```

**의존 방향 규칙**
- 뷰(Popover/Settings/Palette)는 Store·Invoker·Coordinator에 의존한다. 그 반대는 없다.
- `RegistryStore`(가용 도구) 와 `ConfigStore`(멤버십/순서) 는 분리한다. 가용성과 사용자 선택을 섞지 않는다.
- `Invoker`는 앱 특정 지식 없이 URL/앱 실행만 담당한다. (위성이 늘어도 Invoker는 안 바뀐다)
- 팝오버에 보일 목록 = `RegistryStore`의 가용 도구 ∩ `ConfigStore`에서 enabled인 것.

## 7. 데이터 모델

**SatelliteManifest** (위성 앱이 제공하는 파일)
- `id`: 위성 고유 식별자 (예: `"runlet"`)
- `displayName`: 표시 이름
- `urlScheme`: 위성의 URL scheme (예: `"runlet"`)
- `iconRef`: 아이콘 참조 (SF Symbol 이름 또는 리소스 경로)
- `actions`: `[SatelliteAction]`

**SatelliteAction**
- `id`: 액션 식별자
- `title`: 표시 문구 (예: `"배포 실행"`)
- `invoke`: 실행 URL (예: `"runlet://run/deploy"`)
- `keywords`(옵션): 팔레트 검색 보조 키워드
- `iconRef`(옵션): 액션 개별 아이콘

**HubConfigEntry** (허브가 소유하는 위성별 설정)
- `satelliteId`
- `enabled`: 허브 포함 여부 (커스텀 토글의 저장값)
- `pinned`: 상단 고정 (Runlet은 기본 true)
- `sortIndex`: 정렬 순서

**HubConfig**
- `entries`: `[HubConfigEntry]`
- `paletteHotkey`: 팔레트 호출 단축키 표현
- 기타 전역 설정

**SatelliteRuntimeState** (실행 중 계산, 저장 안 함)
- `id`, `isRunning`(실행 여부), `isManagedByHub`(허브 관리 중=아이콘 숨김 대상)

**공유 규약 파일** (위성 앱이 시작 시 확인)
- 위성은 시작할 때 "이 id가 허브에 등록되어 관리 중인가?"를 공유 저장소에서 읽고, 관리 중이면 자기 `NSStatusItem` 생성을 건너뛴다.

## 7+. 위성 앱 menub 호환 계약

> menub는 런처형이라 위성 앱이 아래 계약을 지켜야만 팝오버·팔레트에 나타나고 기능이 호출된다. 새 개인 앱을 만들 때마다 이 체크리스트를 기준으로 삼는다. 앱을 새로 짜는 게 아니라 기존 앱에 세 겹만 덧대는 일회성 작업이다.

**위성 앱이 갖춰야 할 3가지 (체크리스트)**

1. **매니페스트 기록** — 앱 실행 시 `~/Library/Application Support/menub/manifests/<id>.json`에 자기 정보와 액션 목록을 최신본으로 써둔다. (menub가 이 폴더를 읽어 후보로 띄움)
2. **URL scheme 핸들러** — `Info.plist`의 `CFBundleURLTypes`에 자기 scheme 등록 + `onOpenURL`로 수신해, 들어온 URL에 맞는 기능을 실행한다.
3. **아이콘 숨김 체크** — 실행 시 공유 규약 파일에서 "내가 menub에 등록·관리 중인가"를 확인하고, 관리 중이면 자기 `NSStatusItem`을 만들지 않는다. 해제되면 다시 만든다.

**매니페스트 형태 (구체 예시)**

```json
{
  "id": "runlet",
  "displayName": "Runlet",
  "urlScheme": "runlet",
  "iconRef": "sf:terminal",
  "actions": [
    { "id": "deploy", "title": "배포 실행", "invoke": "runlet://run/deploy" },
    { "id": "build",  "title": "빌드",      "invoke": "runlet://run/build" }
  ]
}
```

- `iconRef`는 SF Symbol(`"sf:<name>"`) 또는 리소스 경로. 액션별 `iconRef`·`keywords`는 선택.
- `invoke`의 URL은 그 앱의 scheme으로 시작해야 하며, 앱이 그 경로를 §계약-2의 핸들러에서 해석한다.
- 기능이 늘면 `actions`만 늘리면 되고, menub는 다음 읽기 때 자동 반영한다. (§8-4)

**계약을 지키지 않은 앱의 동작**
- 매니페스트만 있고 URL 핸들러가 없으면: 팝오버에 뜨지만 액션 실행이 실패한다.
- 매니페스트가 없으면: 후보 목록에 아예 뜨지 않는다. (menub 비호환 앱)

## 8. 핵심 기술 챌린지 & 해결 방향

**8-1. 전역 단축키(팔레트 호출)**
- 문제: SwiftUI에 앱이 비활성일 때도 먹는 깔끔한 공개 전역 단축키 API가 마땅치 않다.
- 방향: Carbon `RegisterEventHotKey` 또는 전역 이벤트 모니터로 구현. 방식과 접근성 권한 필요 여부를 구현 단계에서 확정하고 문서에 반영. **추측 시그니처를 코드/문서에 넣지 않는다.**

**8-2. 위성이 자기 메뉴바 아이콘 숨기기**
- 문제: 위성 앱이 "지금 허브가 나를 관리하니 아이콘을 만들지 말자"를 알아야 한다.
- 방향: 공유 규약 파일/플래그를 위성이 시작 시 확인. 이미 실행 중이던 위성은 즉시 반영이 안 될 수 있음 → 재시작으로 반영하거나, 라이브 반영이 필요하면 위성이 플래그 변경을 관찰(파일 감시/분산 알림)하도록 규약 확장. **첫 목표는 재시작 반영으로 단순하게.**

**8-3. URL 호출 시 위성 자동 실행 타이밍**
- 문제: 위성이 안 떠 있을 때 URL만 열면 실행+수신 타이밍이 어긋날 수 있다.
- 방향: `Invoker`가 위성 실행 상태를 먼저 확인 → 미실행이면 앱 실행 후 짧게 대기/재시도하고 URL 전달. 안정화되면 단일 호출로 축소.

**8-4. 매니페스트 동기화 (기능이 바뀔 때)**
- 문제: 위성 기능이 추가/변경되면 허브 목록이 어긋난다.
- 방향: 위성이 **실행될 때마다 자기 매니페스트를 최신본으로 기록**하는 규약. 허브의 `RegistryStore`는 폴더를 감시해 자동 반영. (수동 등록 대안과의 핵심 차이점)

**8-5. 샌드박스 vs 런처 권한**
- 문제: 샌드박스에서는 임의 앱 실행·공유 폴더 접근이 제한된다.
- 방향: 개인용이므로 **비샌드박스**를 기본으로 채택하고 트레이드오프 명시. App Store 배포로 방향이 바뀌면 App Group·보안 스코프 북마크로 재설계 필요.

**8-6. 팝오버 포커스/닫힘과 팔레트 공존**
- 문제: `MenuBarExtra` window 스타일은 포커스 상실 시 닫히므로, 팔레트/설정과의 상호작용에서 의도치 않게 닫힐 수 있다.
- 방향: 팔레트는 별도 패널로 분리하고, 팝오버와 팔레트가 서로를 닫지 않도록 포커스 흐름을 초기부터 검증(M5에서 집중 확인).

## 9. 개발 로드맵 (실행·검증 가능한 마일스톤)

각 마일스톤은 "무엇이 되는지 / 어떻게 켜서 확인하는지 / 어떤 모습인지"로 검증한다.

**M1 — 단일 아이콘 & 빈 팝오버**
- 됨: 메뉴바에 menub 아이콘 1개가 뜨고, Dock 아이콘은 없다. 아이콘 클릭 시 빈 팝오버가 열린다.
- 확인: 빌드·실행 → 메뉴바 우측에 아이콘 확인, 클릭해 팝오버 여닫기.

**M2 — 위성 목록 표시 (더미 매니페스트)**
- 됨: 공유 폴더의 매니페스트를 읽어 팝오버에 도구 그리드로 표시. **Runlet이 맨 위 고정.**
- 확인: 더미 매니페스트 2~3개를 폴더에 넣고 실행 → 팝오버에 목록이 뜨고 Runlet이 상단인지 본다.

**M3 — 액션 2뎁스 & 실제 호출**
- 됨: 위성 선택 → 액션 목록으로 전환 → 액션 클릭 시 URL 호출로 **실제 위성 기능이 실행**된다.
- 확인: Runlet의 한 액션을 눌러 실제 동작(예: 명령 실행)이 일어나는지 관찰.

**M4 — 허브 포함 토글 & 위성 아이콘 숨김**
- 됨: 설정에서 위성별 허브 포함을 켜고 끌 수 있다. 켜면 그 위성이 **자기 메뉴바 아이콘을 숨긴다.**
- 확인: 위성 1개에 규약을 적용 → 허브에서 토글 on → 위성 재실행 시 그 위성 아이콘이 사라지고 허브에만 남는지 본다. 토글 off → 위성 아이콘 복귀 확인.

**M5 — 검색 팔레트**
- 됨: 전역 단축키로 팔레트가 뜨고, 전 위성 액션을 통합 검색·엔터 실행한다.
- 확인: 단축키 → 팔레트 등장 → "배포" 타이핑 → 필터 → 엔터 실행. 팝오버와 서로 안 닫히는지 확인(§8-6).

**M6 — 상태 표시 & 마감**
- 됨: 위성 실행 중 여부 표시, 핀/정렬 커스텀, 전반 다듬기.
- 확인: 실행 중 위성에 표시가 뜨는지, 정렬 변경이 저장·반영되는지 본다.

## 10. cowork 협업 규칙 (바이브코딩 기준)

- **자율 구현**: 협업자(Claude)가 구현을 자율적으로 완성한다. 사용자에게 코드 이해를 요구하지 않는다.
- **보고 형식**: 코드 내부가 아니라 **관찰 가능한 동작** 중심으로 보고한다 — (1) 무엇이 되는지, (2) 어떻게 켜서 확인하는지(빌드·실행 방법, 클릭/단축키 경로), (3) 어떤 모습인지.
- **마일스톤 단위**: 위 M1~M6처럼 실행·검증 가능한 상태로 끊어 보고한다. 중간에 반쯤 된 것을 넘기지 않는다.
- **확정 명세 보호**: §2 확정 표는 임의 변경 금지. 바꿀 이유가 보이면 **트레이드오프와 함께 합의를 먼저** 받는다.
- **열린 결정 처리**: §2 위임 표는 채택한 기본값으로 진행하되, **무엇을 왜 골랐는지 보고**한다. 자잘한 결정으로 흐름을 끊지 않는다.
- **API 정확성**: 추측한 API/시그니처를 코드·문서에 넣지 않는다. 불확실하면 공식 문서로 확인 후 작성하고, **버전 한정 API는 가용성을 명시**한다.
- **설명 용어**: Swift / SwiftUI / AppKit 용어로만 설명한다. **React/JavaScript 비유·예시 금지.**
- **명칭**: 제품명 **"menub"**. 필요 시 교체 가능.

## 11. 부록 — 핵심 API / 빠른 참조

| 용도 | 참조 | 비고 |
|---|---|---|
| 단일 상태 아이템 + 팝오버 | SwiftUI `MenuBarExtra`, `MenuBarExtraStyle.window` | macOS 13+ |
| (대안) 상태 아이템 직접 제어 | AppKit `NSStatusBar.system.statusItem(withLength:)` | 세밀 제어 필요 시 |
| Dock 아이콘 숨김 | `Info.plist` → `LSUIElement` (Application is agent) | — |
| 위성 URL scheme 등록 | 위성 `Info.plist` → `CFBundleURLTypes` | 위성 쪽 |
| URL 수신 | SwiftUI `onOpenURL` (또는 앱델리게이트 open URLs) | macOS 11+ |
| URL 열기(액션 호출) | `NSWorkspace.shared.open(_:)` | — |
| 위성 앱 실행 | `NSWorkspace.shared.openApplication(at:configuration:completionHandler:)` | macOS 11+ |
| 매니페스트/설정 저장 | `~/Library/Application Support/menub/` (또는 App Group) | 비샌드박스 전제 |
| 전역 단축키(팔레트) | Carbon `RegisterEventHotKey` 또는 전역 이벤트 모니터 | **구현 시 방식·권한 확정 (추측 금지)** |
